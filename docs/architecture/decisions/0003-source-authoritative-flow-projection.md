# ADR-0003：采用源码权威的自由流程投影

## 状态

Accepted — 2026-07-12；计划于 2026-08-12 复核。

## 背景

M0–M5 已把任意受支持 C 文件无损投影为语法积木，并把 `main.c` 固定为唯一事实源。M6 要增加自由坐标、端口和电线，但任意 C 中仍可能出现宏、`goto`、解析恢复或当前 CFG 未建模的控制流。若同时持久化一套独立可执行图，源码与图会在编辑、外部修改或恢复模式下分叉。

主要约束是：

- 必须保留任意 C 的原始文本、注释和保守降级行为。
- 节点视觉位置应完全自由，但视觉顺序不得偷偷改变程序语义。
- 所有语义写操作仍要经过旧快照校验、候选补丁、重解析和后置条件。
- 旧项目升级失败时不得损坏 `main.c`。

## 决策

我们采用源码权威的 `FlowProjection`：

- `SourceDoc + ProgramAnalysisSnapshot` 派生节点、端口和控制边；Start/End 映射 CFG 边界。
- `flow-view.json` 只保存坐标、视口、草稿、Checkpoint、面板布局和源码指纹，不保存另一套权威控制图。
- 拖动节点只改变布局。用户连接端口时产生 `ConnectionIntent`；只有候选 C 经完整解析并出现预期 CFG 后置条件后，连接才提交。
- 普通顺序节点最多一个控制输出；只有语法上支持分支的节点可以扇出。
- `raw`、宏边界和 partial CFG 节点可见但锁定拓扑。数据依赖边由 def-use 推导并保持只读。
- 从预设拖入的未连接节点是显式草稿，不参与编译或运行。

## 考虑过的方案

### 图成为唯一事实源

- 优点：节点编辑器实现直接，任意重连容易表达。
- 缺点：无法保证宏、格式、注释和复杂 C 的无损往返；外部源码修改需要脆弱的双主同步。
- 未采用原因：违反现有无损导入契约，并可能让 UI 展示的程序与实际编译源码不同。

### 导入项目与图项目完全分离

- 优点：模型最简单，图项目可使用专用 DSL。
- 缺点：导入任意 C 后不能继续组装，用户必须在两套工作流间迁移。
- 未采用原因：不满足同一工作台中查看、编辑和设计算法的目标。

## 后果

### 正面

- 自由画布不会牺牲源码保真或 M5 的保守语义。
- sidecar 可删除或回滚而不影响程序。
- 新端口类型和算法元素可通过版本化投影扩展。

### 负面

- 任意电线不能立即成立；连接规划必须生成并验证候选 C。
- partial/raw 区域的编辑能力会低于完整 CFG 区域。
- 源码变化后需要进行保守锚点恢复，歧义布局可能被重置。

## 迁移与回滚

旧 v1 项目原样打开，首次保存布局时才创建 sidecar。sidecar 校验失败或源码指纹不匹配时，应用丢弃歧义视图状态并重新投影；绝不据此改写源码。发布回滚只需切回旧搭建面板或忽略 sidecar。

## 架构适应度函数

- 依赖门禁禁止 `src/flow/` 导入 UI、Electron 或核心写路径。
- 金样本验证所有 CFG edge kind、同端点不同类型边和 partial 锁定。
- 工作区测试证明损坏或旧版本 sidecar 不会改写 `main.c`。
