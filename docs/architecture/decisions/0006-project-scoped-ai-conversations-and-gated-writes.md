# ADR-0006：采用项目级 AI 对话与显式授权写入

## 状态

Accepted — 2026-07-14；计划于 2026-08-14 复核。

## 背景

早期 AI 导师只在当前 renderer 内保存一段临时对话，并被架构约束为永久只读。这保证了源码安全，但产生三个产品问题：切换或重开工作区会丢失上下文；同一算法项目不能并行保留“调试”“复杂度”“边界案例”等多批讨论；即使用户明确希望 AI 协助修改程序，也没有一条兼容源码权威模型的受控路径。

本次决策受以下约束驱动：

- 一个托管 workspace opaque ID 必须稳定对应一个 AI Project，重复打开恢复而不是复制。
- 每个 AI Project 需要多对话、重命名、归档、删除和本地持久化。
- renderer 可以读取对话正文和公开元数据，但不得取得 API 密钥、密文、绝对路径或任意文件系统能力。
- `main.c` 继续是唯一源码事实源；AI Project 数据损坏、迁移失败或被删除均不得改写源码。
- AI 修改默认关闭。开启后也不得把模型输出直接当作代码事实或绕过现有解析、CFG、clang、diff、检查点和撤销门禁。
- 本地历史必须有 schema、容量上限、乐观并发和原子写，避免无界增长或静默覆盖。

## 决策

我们采用“一个托管 workspace 对应一个本地 AI Project”的模型：

- AI Project 以 workspace opaque ID 定位，并在该托管条目内保存独立的 `ai-project.json`。首次打开时幂等创建，之后恢复相同 `projectId`。
- 一个 Project 包含最多 64 个对话；对话可创建、读取、重命名、归档和删除。消息按对话持久化，并可选绑定源码 fingerprint 作为证据上下文。
- 存储 schema 从 v1 开始，使用 revision 乐观并发、同条目串行 mutation、同目录临时文件、`fsync` 和原子 rename。单文件上限 4 MiB，每个对话最多 256 条消息，整个 Project 最多 2,048 条消息，单条消息最多 64 KiB。
- preload 只暴露具名的 Project/Conversation 方法；每个请求在主进程验证精确对象形状、opaque ID、revision、标题和内容上限。结果统一为判别联合，预期错误不通过异常穿越 IPC。
- 对话存储不保存 Provider 凭据、请求头、绝对路径或任意 Endpoint。API 密钥继续只存在于现有 OS 加密 Provider store。
- AI 权限分为默认只读、修改前复核与受控执行。模型只能产出严格结构化的候选替换；应用层绑定当前 workspace、revision 与 fingerprint，生成精确 diff，建立检查点，通过源码边界、重解析、无损往返与 CFG 后置条件，再经现有源码编辑管线提交并进入 CodeMirror 撤销历史。复核模式还必须由用户在工作台内显式确认。
- `src/ai/` 继续禁止导入核心写接口。AI 不参与 C 解析、类型检查、语义等价裁定或算法标签生成。

本 ADR 已落地 Project/Conversation 后端、IPC、非模态工作台窗口与上述受控源码修改链。默认权限仍为只读；任何路径不明确、锚点不唯一、源码已变化、投影为 raw/partial/locked、解析失败、无损往返失败或 CFG 退化的候选修改都拒绝生效。当前不开放文件创建/删除、工作区外写入、命令执行或任意路径能力。

## 考虑过的方案

### 方案 A：全应用共用一段聊天历史

- 优点：存储和 UI 最简单，模型容易获得长上下文。
- 缺点：不同课程项目互相污染；删除或导出单个项目时无法界定数据归属；容易把错误项目的代码证据发送给模型。
- 未采用原因：不满足按项目分类与多批对话，且扩大上下文误用风险。

### 方案 B：每个窗口只保存内存会话

- 优点：不需要迁移、磁盘 schema 或容量治理；关闭窗口即清除隐私数据。
- 缺点：重启、崩溃或切换工作区即丢失学习记录；同一项目无法形成长期可追踪的讨论。
- 未采用原因：与“重复打开项目恢复原 Project”的明确产品需求冲突。

### 方案 C：Renderer 使用 localStorage 保存对话

- 优点：无需新增主进程 IPC，开发成本低。
- 缺点：难以原子写、做 revision 冲突和逐项目容量控制；renderer 可篡改整个 store，数据也不随 Documents 托管条目备份。
- 未采用原因：不符合既有 Documents 工作区、安全边界和恢复模型。

### 方案 D：开启总开关后让 AI 直接写 `main.c`

- 优点：操作次数最少，自动化程度最高。
- 缺点：模型输出可绕过源码 revision、diff 和语法/CFG 验证；失败时难以归因或撤销，也违反源码权威决策。
- 未采用原因：不可接受的数据损坏与教学误导风险。

## 后果

### 正面

- AI 对话与用户真正操作的工作区拥有稳定的一一归属，不会跨项目串线。
- 多批对话可以分别承载调试、解释、优化和案例设计，重启后仍可恢复。
- 存储损坏、版本冲突或容量超限会 fail closed，且完全不触碰 `main.c`。
- AI 修改复用既有源码权威和验证基础，没有引入第二套写入事实源。
- Renderer 仍只看到 opaque ID、公开元数据和用户需要的聊天正文。

### 负面

- 每个消息 mutation 当前会原子重写有界 JSON 文件；4 MiB 范围内可接受，但不适合无限历史。
- 对话删除是硬删除；如需回收站，需要新的保留期和恢复决策。
- 乐观并发要求 UI 在冲突后重新载入，不能静默“最后写入者获胜”。
- 受控 AI 改码增加了权限状态、候选 diff、检查点与验证控制器，维护成本高于只读对话。

### 中性

- AI Project 跟随托管 workspace 条目保存，而 Provider 凭据仍保存在 OS 加密的全局 userData store；两者生命周期有意分离。
- 归档只改变可见状态，不减少容量；需要释放容量时用户必须删除对话。

## 迁移与回滚

- 现有 workspace 没有 `ai-project.json` 时，首次打开 AI Project 才创建 v1 文件；不扫描或改写 `main.c`。
- 旧的 renderer 内存聊天没有可靠项目归属，因此不做猜测性迁移。
- 未知 schema、损坏 JSON、符号链接或超限文件一律保留原文件并返回具名失败，不自动覆盖。
- 回滚应用版本时，旧版本会忽略额外的 `ai-project.json`；源码和其他 sidecar 仍可正常使用。
- 若需要关闭该功能，只需停止注册 AI Project IPC；删除或保留聊天文件均不影响编译、投影和运行。

## 架构适应度函数

- Store 单元测试证明重复 `open` 恢复同一 `projectId`，并覆盖多对话 CRUD、消息持久化、revision 冲突、并发串行和容量限制。
- 文件系统测试证明恶意 ID、额外字段、符号链接、损坏或超限 store 全部 fail closed，且 `main.c` 逐字不变。
- IPC 合同测试锁定七个具名操作、主 frame 授权、严格 arity 和 shutdown 失败语义。
- Preload/Electron 边界测试断言没有 path、通用文件系统、credential、decrypt 或通用 IPC 通道暴露给 renderer。
- AI 修改验收证明：默认关闭；每批修改绑定 workspace/revision/fingerprint；具有 diff、检查点、确定性验证和撤销；任何一步失败不得写入源码。
