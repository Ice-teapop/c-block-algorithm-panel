# ADR-0005：分离实测效率与复杂度结论

## 状态

Accepted — 2026-07-12；计划于 2026-08-12 复核。

## 背景

用户需要每次运行后的时间、内存和优化帮助。单次墙钟时间受机器、输入、工具链和系统负载影响，不能推出 Big-O；静态循环层数也不能在未知输入规模下自动等同算法复杂度。把不同证据压成一个“效率分数”会产生精确但错误的教学结论。

## 决策

- 每次运行分别记录编译时间、墙钟时间、峰值 RSS、峰值进程数、输出字节、执行事件数、操作计数和终止原因。
- 只有源码指纹、场景、工具链和测试输入一致的记录才能直接比较。
- Benchmark 使用多个输入规模、预热和重复样本，中位数与操作计数增长曲线只作为经验增长率。
- Big-O 结论必须来自明确的静态证据或用户声明，并携带置信度与适用条件。
- 本地导师提示必须引用对应节点、finding 或运行记录；不生成综合效率分数，也不自动修改代码。
- 教学模拟结果与真实运行历史分库存放，模拟结果不得参与真实性能比较。

## 考虑过的方案

### 单一 0–100 效率分数

- 优点：易懂、易排序。
- 缺点：权重任意，混淆机器性能、输入和渐近复杂度，容易误导初学者。

### 只显示原始耗时

- 优点：完全客观，实现简单。
- 缺点：缺少上下文、基线和教学解释，无法帮助用户判断改动是否有效。

## 后果

### 正面

- 每个结论都能追溯到真实证据。
- 运行历史可以安全比较而不夸大精度。
- 未来 AI Provider 必须遵循同一证据契约。

### 负面

- UI 需要同时解释多种指标和置信度。
- 获得经验增长率需要多次运行，不能在一次普通运行后承诺复杂度。

## 迁移与回滚

旧运行结果没有持久化，无需数据迁移。`run-history.json` 使用独立 schema；损坏时只丢弃历史，不影响源码或运行器。关闭历史功能后仍保留单次原始结果。

## 架构适应度函数

- 类型系统区分 `measured`、`static` 与 `simulated` 证据。
- 测试拒绝跨场景或跨工具链直接比较。
- UI 契约测试禁止把模拟记录呈现为真实基线。
