# ADR-0004：采用有界影子源码运行轨迹

## 状态

Accepted — 2026-07-12；计划于 2026-08-12 复核。

## 背景

自由流程画布需要实时显示实际执行路径。直接修改项目源码会污染事实源；调试器附加会显著扩大权限、平台差异和不可信代码风险；无界事件推送又可能让恶意或错误程序耗尽 renderer 内存。

## 决策

- Trace 使用当前源码指纹生成临时影子源码并插入有界事件记录，不修改 `main.c`。
- renderer 只能调用 `startTrace`、按序号读取 `readTrace` 和 `cancelTrace`，不能获得路径或原生进程句柄。
- session 绑定发起窗口、源码指纹、一次性运行授权和资源限制。
- renderer 约每 100 ms 拉取增量批次；总计最多 10,000 个事件或 8 MiB，超限后明确标记截断。
- 源码改变、窗口销毁、超时或取消都会使 session 失效并终止整组进程。
- raw、宏、partial CFG 或无法证明插桩位置安全的函数不提供完整轨迹；普通编译运行仍可继续。
- “运行”只编译并执行一次当前源码；影子 Trace 只能由独立“观察”动作启动，Benchmark 也不会为每个样本隐式启动 Trace。
- Pause 只暂停轨迹回放和教学模拟，不以 `SIGSTOP` 挂起真实原生程序。

## 考虑过的方案

### 直接推送每个运行事件

- 优点：延迟最低。
- 缺点：跨 IPC 洪泛难以施加背压，旧窗口和旧源码事件容易污染当前会话。

### 集成 LLDB 逐步调试

- 优点：无需源码插桩，可获得底层调试状态。
- 缺点：权限、进程控制、平台绑定和用户心智成本远超本科算法流程可视化需要。

## 后果

### 正面

- 项目源码与运行轨迹严格隔离。
- 批次读取提供背压、顺序和确定的资源上限。
- 现有 runner 的信任确认和进程组清理可继续复用。

### 负面

- 轨迹刷新不是逐指令级，最坏会有一个轮询周期延迟。
- 某些宏、`goto` 或表达式级控制流只能显示部分轨迹。
- 插桩构建增加一次独立编译成本。

## 迁移与回滚

Trace 是独立 IPC 能力；关闭能力标志后，运行器退回现有 compile/run/diagnose，不影响源码和制品。所有临时目录沿用 runner 的过期与退出清理。

## 架构适应度函数

- session、事件数、字节数、墙钟和进程数上限都有单元测试。
- E2E 验证源码变化、取消和窗口关闭后旧事件不会呈现。
- preload 仅暴露不可伪造的 opaque session ID。
